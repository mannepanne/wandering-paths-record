# D1 database setup

Cloudflare D1 (SQLite) is the database for Wandering Paths. It runs entirely server-side, bound to the Cloudflare Worker — there is no client-side database SDK.

**Authoritative schema:** `scripts/d1-schema.sql`

**Database name:** `wandering-paths`
**Database ID:** `f928b168-fb21-4420-bff8-2f9320d3f22e`
**Worker binding:** `env.DB`

---

## Schema overview

Three tables. Schema is SQLite, not PostgreSQL — key differences from the old Supabase schema are noted below.

### `restaurants` table

Core restaurant data.

| Column | Type | Notes |
|--------|------|-------|
| `id` | TEXT | UUID as text string (no native UUID type in SQLite) |
| `name` | TEXT NOT NULL | |
| `address` | TEXT | Human-readable location summary |
| `website` | TEXT | |
| `status` | TEXT | `'to-visit'` or `'visited'` — enforced with CHECK |
| `personal_appreciation` | TEXT | `'unknown'`, `'avoid'`, `'fine'`, `'good'`, `'great'` — cached from latest visit |
| `description` | TEXT | |
| `visit_count` | INTEGER | Cached count of non-placeholder visits |
| `cuisine` | TEXT | Legacy field, kept for compatibility |
| `cuisine_primary` | TEXT | |
| `cuisine_secondary` | TEXT | |
| `style` | TEXT | |
| `venue` | TEXT | |
| `specialty` | TEXT | JSON array stored as text (no native array type in SQLite) |
| `must_try_dishes` | TEXT | JSON array stored as text |
| `price_range` | TEXT | |
| `atmosphere` | TEXT | |
| `dietary_options` | TEXT | |
| `source` | TEXT | |
| `source_url` | TEXT | |
| `latitude` | REAL | Legacy — prefer `restaurant_addresses` |
| `longitude` | REAL | Legacy — prefer `restaurant_addresses` |
| `public_rating` | REAL | Google Places rating |
| `public_rating_count` | INTEGER | |
| `public_review_summary` | TEXT | AI-generated from Google reviews |
| `public_review_summary_updated_at` | TEXT | ISO datetime string (no TIMESTAMPTZ in SQLite) |
| `public_review_latest_created_at` | TEXT | |
| `created_at` | TEXT | `datetime('now')` default |
| `updated_at` | TEXT | `datetime('now')` default |

**SQLite type notes:**
- Arrays (`specialty`, `must_try_dishes`) are JSON strings — Worker parses them with `JSON.parse()`
- Datetimes are TEXT in ISO format, not PostgreSQL `TIMESTAMPTZ`
- No UUID type — IDs are TEXT, generated by Worker before insert

### `restaurant_addresses` table

Individual locations for multi-location chains (e.g., Dishoom with 8 branches).

| Column | Type | Notes |

| `location_name` | TEXT | e.g., "Shoreditch", "Canary Wharf" |
| `full_address` | TEXT | |
| `city` | TEXT | |
| `country` | TEXT | |
| `latitude` | REAL | |
| `longitude` | REAL | |
| `phone` | TEXT | |
| `created_at` | TEXT | |
| `updated_at` | TEXT | |

### `restaurant_visits` table

Visit history with individual ratings and notes.

| Column | Type | Notes |
|--------|------|-------|
| `id` | TEXT | UUID as text |
| `restaurant_id` | TEXT | Foreign key → `restaurants(id)` ON DELETE CASCADE |
| `visit_date` | TEXT | ISO date string `YYYY-MM-DD` |
| `rating` | TEXT | `'unknown'`, `'avoid'`, `'fine'`, `'good'`, `'great'` |
| `experience_notes` | TEXT | Optional, max 2000 chars |
| `company_notes` | TEXT | Optional, max 500 chars |
| `is_migrated_placeholder` | INTEGER | `0` or `1` (SQLite boolean) |
| `created_at` | TEXT | |
| `updated_at` | TEXT | |

**Constraint:** `UNIQUE(restaurant_id, visit_date)` — one visit record per restaurant per day.

### Indexes

```sql
idx_restaurants_status           -- restaurants(status)
idx_restaurants_appreciation     -- restaurants(personal_appreciation)
idx_restaurants_rating_count     -- restaurants(public_rating_count)
idx_addresses_restaurant_id      -- restaurant_addresses(restaurant_id)
idx_visits_restaurant_id         -- restaurant_visits(restaurant_id)
idx_visits_date                  -- restaurant_visits(visit_date)
idx_visits_restaurant_date       -- restaurant_visits(restaurant_id, visit_date)
```

## Cached rating architecture

`restaurants.personal_appreciation` and `visit_count` are **cached values**, maintained in sync by the Worker after every visit mutation (replaces the PostgreSQL trigger from the Supabase era).

```javascript
// src/worker.js — syncRestaurantVisitData()
// Called after any INSERT, UPDATE, or DELETE on restaurant_visits
async function syncRestaurantVisitData(db, restaurantId) {
  const latest = await db.prepare(
    `SELECT rating FROM restaurant_visits
     WHERE restaurant_id = ?
     ORDER BY visit_date DESC, created_at DESC LIMIT 1`
  ).bind(restaurantId).first();
  // Updates personal_appreciation + visit_count in restaurants table
}
```

This avoids N+1 queries on the restaurant list page — one query fetches all restaurants with their cached rating.

## Worker API endpoints

All data access goes through `src/worker.js`. The D1 binding (`env.DB`) is only available server-side.

### Public routes (no auth required)

| Method | Path | Description |
|--------|------|-------------|
| GET | `/api/restaurants` | All restaurants with locations; supports `?status=`, `?cuisine=`, `?search=` filters |
| GET | `/api/restaurants/meta` | Distinct values for filter dropdowns (cuisines, etc.) |
| GET | `/api/restaurants/:id` | Single restaurant with locations |
| POST | `/api/extract-restaurant` | AI extraction from URL — **now requires auth** |
| POST | `/api/claude` | Claude API proxy |
| GET | `/api/google-maps` | Google Maps proxy (geocoding, places, details) |

### Protected routes (require valid CF_Authorization JWT)

| Method | Path | Description |
|--------|------|-------------|
| GET | `/api/auth/me` | Returns identity from verified JWT |
| GET | `/api/visits/:restaurantId` | Visit history for a restaurant |
| POST | `/api/admin/restaurants` | Create restaurant |
| PUT | `/api/admin/restaurants/:id` | Update restaurant |
| DELETE | `/api/admin/restaurants/:id` | Delete restaurant |
| POST | `/api/admin/restaurants/:id/addresses` | Add address to restaurant |
| PUT | `/api/admin/addresses/:id` | Update address |
| DELETE | `/api/admin/restaurants/:id/addresses` | Clear all addresses |
| DELETE | `/api/admin/addresses/:id` | Delete single address |
| POST | `/api/admin/visits` | Add visit record |
| PUT | `/api/admin/visits/:id` | Update visit |
| DELETE | `/api/admin/visits/:id` | Delete visit |

Auth is checked by reading the `CF-Authorization` cookie and verifying the JWT. See `REFERENCE/auth-setup.md` for details.

## Local development with D1

For local development, the Worker uses a local D1 instance managed by Wrangler.

```bash
# Start Worker in local dev mode (uses local D1, not production)
npx wrangler dev

# Run a schema migration against local D1
npx wrangler d1 execute wandering-paths --local --file=scripts/d1-schema.sql

# Run against production D1
npx wrangler d1 execute wandering-paths --file=scripts/d1-schema.sql

# Query local D1 directly
npx wrangler d1 execute wandering-paths --local --command="SELECT count(*) FROM restaurants"
```

**Note:** The standard frontend dev server (`npm run dev`) still runs on port 8080 and connects to the local API server (`npm run api`) on port 3001 for AI extraction. For full D1 integration locally, run `npx wrangler dev` instead and hit the Worker directly.
